### 二、 你的方案评估与“排雷”

你规划的“LiveKit client-sdk-cpp + 极简 JNI + Kotlin 壳”整体架构是合理的，符合大厂跨平台 SDK 的设计范式。但**有几个致命的隐患，如果在面试中暴露，会直接挂掉**：

#### 🚨 隐患 1：采集和音频处理放在 Kotlin 层（严重减分项）
*你的规划：“权限、摄像头、音频全部Kotlin原生处理，完全不用碰C++。”*
* **面试官视角**：如果 Kotlin 层通过 Camera2 拿到 YUV 数据，通过 JNI 传给 C++ 编码，这中间会产生**巨大的内存拷贝开销和 GC 抖动**。大厂 RTC SDK 的采集、3A（回声消除/降噪）、前处理几乎全在 C++ 层调度。
* **修正方案**：Kotlin 只负责**生命周期和权限申请**，真正的采集必须使用 WebRTC 原生的 `Camera2Capturer`，渲染使用 `SurfaceViewRenderer` / `SurfaceTextureHelper`。**核心原则是：音视频数据流（YUV/PCM）绝对不能进入 Java/Kotlin 堆内存，必须通过 C++ 层的 `VideoTrack` / `AudioTrack` 进行零拷贝流转。**

#### 🚨 隐患 2：“魔改”的深度不够，沦为“调参侠”
*你的规划：“魔改 JitterBuffer/NACK/QoS”*
* **面试官视角**：如果你只是改了 JitterBuffer 的最大延迟阈值，或者调整了 NACK 的重试次数，这叫“调参”，不叫“魔改”。
* **修正方案**：必须深入到算法逻辑。例如：WebRTC 默认的 JitterBuffer (VCMTiming) 在剧烈网络抖动下表现不佳，你是否能引入**基于卡尔曼滤波的延迟预测**？或者针对门禁场景（周期性 I 帧），优化 NACK 和 FEC 的自适应策略？

#### 💡 关于 LiveKit vs Mediasoup
* **强烈建议选择 LiveKit**。LiveKit 的 `client-sdk-cpp` 是基于 libwebrtc 封装的，架构现代，适合做端侧二次开发。Mediasoup 主要强在 SFU 服务端（Node.js/C++），客户端没有现成的 C++ SDK，自己用原生 libwebrtc 拼凑工作量极大，3个月根本做不完。

### 三、 技术亮点设计（如何打动面试官）

3个月时间，不要贪多，做透以下 **2 个核心亮点**，足以让面试官眼前一亮：

#### 亮点 1：基于场景感知的 QoS 动态自适应（NACK/FEC）
* **痛点**：原生 WebRTC 的 FEC（前向纠错）和 NACK（重传）策略相对固定，在带宽骤降时容易导致画面卡死。
* **你的魔改**：结合 TWCC (Transport-wide congestion control) 的反馈，设计一套动态调整算法。
  * 当检测到 RTT 突增且带宽受限时，**动态降低 FEC 冗余度，增加 NACK 权重**。
  * 当丢包率 > 15% 时，触发**降级策略**：优先保音频，视频只请求关键帧（PLI/FIR），放弃非关键帧的 NACK。
* **产出**：画出架构图，写出核心 C++ 代码逻辑。

#### 亮点 2：JitterBuffer 抗抖动优化
* **痛点**：原生 WebRTC 在面临周期性网络抖动（如 Wi-Fi 信道拥挤导致的周期性丢包）时，JitterBuffer 会频繁调整延迟，导致画面忽快忽慢。
* **你的魔改**：在 `modules/video_coding/timing` 模块中，增加对**网络抖动方差的平滑处理**，或者针对特定业务（如你的门禁对讲）设计**低延迟优先的 JitterBuffer 策略**（牺牲一点流畅度，换取极致的低延迟）。

#### 亮点 3：建立专业的弱网测试与 Profiling 体系（必做！）
没有数据的优化都是耍流氓。
1. **弱网模拟**：使用 Linux `tc` (Traffic Control) 或 Mac 的 `Network Link Conditioner`，编写自动化脚本，模拟 20% 丢包、300ms 抖动、500ms 延迟。
2. **数据对比**：输出你的魔改版 vs 原版 LiveKit/WebRTC 的对比图表（**视频卡顿率、端到端延迟、首帧时间**）。
3. **性能分析**：使用 Android Studio Profiler 或 **Perfetto / Systrace**，抓取 C++ 层的线程调度，证明你的魔改没有引入额外的 CPU 开销和锁竞争。

你的规划整体思路非常清晰，**“不用从零手写全套，基于优秀开源项目做深度魔改”** 是业余时间做项目的最优解。
