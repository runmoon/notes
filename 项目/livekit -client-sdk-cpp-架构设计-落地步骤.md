
# 一、分阶段分步落地实施路线（从0到完整可开源项目，4阶段）
## 阶段1：基础环境搭建 + 原生基线版本（1~2周，产出基准对比样本）
目标：搭建可运行原生LiveKit+libwebrtc Android Demo，拿到未优化的性能、弱网基线数据，作为后续对比标准
1. WebRTC编译环境部署
   - 搭建depot_tools，拉取M114分支源码，配置清华镜像同步脚本；
   - 完成GN编译脚本编写，支持Android ARM64动态库编译，关闭测试用例、裁剪无用模块降低库体积；
2. 原生LiveKit C++ Demo打通
   - 实现Camera2采集、MediaCodec硬编、WebRTC发送；远端解码Surface渲染；
   - JNI基础胶水层完成：房间加入、推拉流、音视频回调、生命周期管理；
3. 搭建标准化弱网测试环境
   Linux服务器部署tc+netem弱网脚本，固化测试用例（10%随机丢包、20%周期性丢包、带宽限500kbps三套标准场景）；
4. 采集基线Benchmark数据
   原生版本跑全量弱网用例，记录卡顿率、E2E延迟、JitterBuffer最大缓存；
   使用Systrace/Perfetto抓取10分钟通话，统计CPU、PSS内存、GC停顿次数，填入对比表格。

## 阶段2：核心QoS模块魔改落地（3~4周，项目核心价值产出）
按优先级逐个实现文档中三大底层优化，每完成一个模块单独做对比测试
### Step2.1 JitterBuffer周期性抖动优化（优先级最高）
1. 定位源码：`modules/video_coding/timing/vcm_timing.cc`、`frame_buffer.cc`；
2. 实现滑动窗口抖动方差计算模块，区分随机抖动/周期性抖动特征；
3. 动态平滑因子逻辑：周期性抖动时抬高缓冲平滑系数，限制渲染速率剧烈波动；
4. 单独跑周期性丢包测试，对比原生JitterBuffer延迟、画面平滑度指标。

### Step2.2 动态NACK/FEC协同控制器
1. 解析TWCC反馈报文，提取RTT、逐包丢包序列；
2. 实现Gilbert丢包预测模型，根据实时网络状态动态调整FEC冗余比例；
3. 实现NACK降级逻辑：RTT>200ms+丢包>15%时，放弃非I帧重传、放大RTO超时；
4. 修改rtp_sender_video发送逻辑，绑定自研控制器；
5. 随机丢包场景测试，对比卡顿率、音频丢包次数。

### Step2.3 PLI/FIR关键帧请求节流器
1. 新增独立节流管理类，记录带宽预估、当前GOP大小；
2. 带宽不足时延迟PLI发送，优先依靠FEC恢复丢失帧；
3. 限制单位时间内最大关键帧请求次数，杜绝请求风暴；
4. 带宽骤降场景专项测试，统计恢复首帧耗时。

### Step2.4 统一埋点体系
在所有自研QoS模块插入Perfetto跟踪埋点，输出抖动方差、FEC比例、带宽预估、NACK计数等监控指标。

## 阶段3：架构性能优化 + 零拷贝改造（2周，补齐方案短板）
目标：解决Java层拷贝、GC卡顿，落地Roadmap规划的MediaCodec深度优化
1. Native层YUV/PCM零拷贝重构
   Camera2采集直接输出AHardwareBuffer，不经过Java ByteBuffer；C++层直接持有native buffer指针，全程不进入Java堆；
2. MediaCodec硬编硬解AHardwareBuffer直通
   改造编码/解码链路，Native直接操作硬件缓冲区，删除CPU侧YUV拷贝流程；
3. SurfaceTexture渲染直通优化
   C++层绑定Surface，JNI仅传递Surface指针，解码纹理直接输出到渲染层，绕过内存拷贝；
4. 优化JNI调用频次，批量回调，减少Java/Native切换开销；
5. 重新全量跑分，更新Benchmark性能表格，体现内存、CPU、GC优化收益。

## 阶段4：音频3A优化 + 拓展能力 + 开源文档完善（2~3周，项目完整闭环）
### Step4.1 音频模块针对性优化（IoT对讲核心）
1. 改造WebRTC AEC3，针对户外嘈杂、远距离对讲场景优化非线性回声抑制；
2. 适配窄带宽OPUS参数，动态调整音频码率、帧长，弱网优先保障音频；

### Step4.2 拓展预研（Roadmap预留能力，可选实现）
- 轻量SVC H.265分层编码适配；
- QUIC/KCP传输层替换小模块实验（可选，作为技术拓展亮点）；

### Step4.3 多边界场景补充测试
新增4G/Wi-Fi切换、断网重连、多路并发对讲、高RTT(500ms+)弱网测试，完善测试报告；

### Step4.4 项目文档、工程脚本、开源标准化
1. 补齐所有TODO占位内容：算法公式、源码路径、阈值说明；
2. 完善构建脚本、一键弱网测试脚本；
3. 补充架构图、Systrace性能截图、弱网延迟对比折线图；
4. 完善README、Roadmap、开源协议、使用Demo教程，整理成可直接开源完整仓库。

# 三、落地关键避坑点
1. WebRTC源码修改最小侵入原则
   尽量用子类/外部控制器包裹原生逻辑，少直接修改原生函数内部代码，降低后续版本升级冲突；
2. 阈值参数做可配置化
   抖动平滑系数、RTT/丢包阈值、PLI冷却时间全部抽成全局配置，适配不同算力IoT设备（门禁/监控）；
3. 分模块灰度验证
   每完成一个优化模块单独测试，不要一次性合并所有改动，出现性能退化便于定位问题；
4. ARM低算力设备优先验证
   不要只在高端手机测试，用树莓派、低性能门禁ARM板跑压测，贴合项目IoT定位。







这份分阶段实施路线**非常专业、逻辑严密，且极具实战价值**。你精准地抓住了大厂面试官看重的核心要素：**基线对比、数据驱动、底层深度、以及针对特定场景（IoT）的业务思考**。避坑点总结得也极其到位，特别是“最小侵入原则”和“低算力设备验证”，这体现了成熟工程师的素养。

但在实际落地前，我需要从**工程现实**的角度为你做一次“深度体检”，并为你提供更细化的“保姆级”执行手册。

### 一、 整体分析与潜在风险预警（必读）

#### 1. 时间评估过于乐观（最大风险）
* **阶段1（1~2周）**：如果你是第一次编译 libwebrtc，`depot_tools` 和 `gclient sync` 的玄学问题（网络、依赖冲突、Python版本）可能会消耗你 1 周时间。**建议预留 3 周**。
* **阶段3（2周）**：**严重低估了 Android 零拷贝的难度**。`AHardwareBuffer` 与 WebRTC 的 `VideoFrame` 体系融合，涉及不同手机厂商 MediaCodec 的奇葩实现、生命周期管理、纹理格式转换（YUV vs RGB）。2周极难完成，建议拉长到 3-4 周，或降低预期（先做半零拷贝）。

#### 2. 技术难点与“造轮子”陷阱
* **Gilbert 丢包模型**：学术界很好，但在 C++ 工程中实现并调参极难。大厂通常用更简单的启发式规则（如基于 EWMA 指数加权移动平均）。**建议：先用简单的滑动窗口+阈值判断，跑通后再上复杂模型。**
* **WebRTC 版本差异**：你提到 M114。注意，M114 的 JitterBuffer 已经从旧的 `FrameBuffer` 升级到了基于 `FrameBuffer3` (或类似新架构) 的实现，源码结构和老版本（如 M90 以前）完全不同，看博客容易迷路，**必须以你拉取的源码为准**。

#### 3. 阶段顺序微调建议
* **建议将“阶段3（零拷贝架构）”的部分工作前置**。如果在阶段2（QoS魔改）时，数据还在 Java 层拷贝，你测出来的 CPU 和延迟基线是不准的。
* **优化方案**：阶段1先跑通“能用的原版”；**阶段2初期，先花几天把基础的 Native 采集/渲染直通搞定**，确立真正的“性能基线”，然后再开始改 QoS 算法。
